# TikTok Connector Spec for Gluebox

## Goal

Add a `tiktok` platform to the `stonkwatch_social` connector so that ASX announcement content is auto-posted to TikTok as short-form video. This follows the same pattern as the existing X, Bluesky, Meta, and Threads platforms.

## Why Video (Not Just Images)

TikTok only accepts video content. Unlike Instagram/Facebook where we can post a static image card, TikTok requires a video file. The approach: **generate a short video from the existing OG image card + text overlay using FFmpeg**, then upload via the Content Posting API.

## TikTok Content Posting API

### Authentication

- OAuth 2.0 with scopes: `video.publish`, `video.upload`
- Access token in Authorization header: `Bearer {access_token}`
- Tokens expire; refresh flow same pattern as X connector

### Upload Flow (3 steps)

**Step 1: Initialise upload**
```
POST https://open.tiktokapis.com/v2/post/publish/video/init/
Authorization: Bearer {access_token}
Content-Type: application/json

{
  "post_info": {
    "title": "$BHP iron ore output up 12% â€” full AI breakdown on stonkwatch.app",
    "privacy_level": "PUBLIC_TO_EVERYONE",
    "disable_duet": false,
    "disable_stitch": false,
    "disable_comment": false
  },
  "source_info": {
    "source": "FILE_UPLOAD",
    "video_size": <byte_count>,
    "chunk_size": <byte_count>,
    "total_chunk_count": 1
  }
}
```

Response: `{ "data": { "publish_id": "...", "upload_url": "..." } }`

**Step 2: Upload video bytes**
```
PUT {upload_url}
Content-Range: bytes 0-{size-1}/{size}
Content-Type: video/mp4

<video bytes>
```

**Step 3: Check publish status**
```
POST https://open.tiktokapis.com/v2/post/publish/status/fetch/
Authorization: Bearer {access_token}

{ "publish_id": "..." }
```

Response includes `status`: `PROCESSING`, `PUBLISH_COMPLETE`, or error.

### Rate Limits
- 6 requests per minute per user token
- Build in backoff between posts (the existing 5-second delay between platform posts handles this)

### App Registration
- Register at developers.tiktok.com
- App type: "Content Posting"
- Request `video.publish` and `video.upload` scopes
- Goes through TikTok app review (similar to Meta App Review)

## Video Generation (video.rs)

### Approach: FFmpeg from Static Images

Use the existing OG card (1200x630) and story card (1080x1920) PNGs generated by `og_image.rs` as source material.

```rust
pub async fn generate_tiktok_video(
    story_image_path: &Path,       // 1080x1920 story card
    social_text: &str,             // AI-generated caption
    output_path: &Path,
    duration_secs: u64,
) -> Result<PathBuf>
```

### FFmpeg Command

```bash
ffmpeg -loop 1 -i story-card.png \
  -vf "zoompan=z='min(zoom+0.001,1.08)':d={fps*duration}:x='iw/2-(iw/zoom/2)':y='ih/2-(ih/zoom/2)':s=1080x1920,format=yuv420p" \
  -c:v libx264 -t {duration} -pix_fmt yuv420p -r 30 \
  output.mp4
```

This creates a subtle Ken Burns zoom-in effect on the story card, producing a clean, professional-looking TikTok video from a single image.

### Dependencies
- `ffmpeg` must be available in PATH (add to nix flake)
- `tokio::process::Command` for async execution
- Re-uses existing Typst-generated story cards â€” no new image generation needed

### Caption / Title

TikTok `post_info.title` has a 150-character limit. The existing pipeline.rs generates 280-char social copy for X. For TikTok, either:
- Truncate to 150 chars (simplest)
- Add a TikTok-specific prompt variant in pipeline.rs (better quality)

Recommendation: add a `tiktok_caption()` method to `PostCandidate` that truncates or reformats.

## Registration & OAuth Flow

### One-Time Setup (Manual)

1. Go to developers.tiktok.com â†’ Create App â†’ Content Posting type
2. Add product: "Content Posting API"
3. Request scopes: `video.publish`, `video.upload`
4. Set redirect URI (e.g., `http://localhost:8990/api/tiktok/callback`)
5. Submit for review
6. Once approved, run the OAuth flow to get access + refresh tokens
7. Add tokens to `gluebox.toml`

### OAuth Callback Endpoint

Add a new webhook endpoint in gluebox:

```
GET /api/tiktok/callback?code={auth_code}
```

Exchanges the auth code for access + refresh tokens and stores them. This only needs to run once during initial setup.
